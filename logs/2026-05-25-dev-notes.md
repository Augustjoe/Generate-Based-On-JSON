## 项目目标

当前项目方向是做一个让开发者更省力的 Naive UI Admin 底座。短期目标不是直接做完整低代码平台，而是先把后台管理系统里高频、重复的能力沉淀好，例如布局、菜单、登录、权限、表单、表格、ProTable、编辑态配置和常用业务组件。

更长期的愿景是：开发者可以通过稳定的 JSON/schema 配置快速生成表单、表格和业务页面；后续再逐步发展到前端可视化配置、配置保存和页面生成。

## 今天完成的主要改动

### 1. 生产环境关闭 Vue DevTools

修改文件：

### 2. 路由守卫日志降级

修改文件：

内容：

- 路由守卫日志改为仅开发环境输出。
- 不再打印 token 和完整 userInfo，避免敏感信息泄露。
- 只输出目标路径、是否已登录、是否存在 userInfo。

### 3. Naive UI 动态组件加载改为 `import.meta.glob`

修改文件：

- `src/utils/dynamicComponent.ts`

内容：

- 原先通过字符串拼接 `import('../../node_modules/naive-ui/es/${name}/index.mjs')` 动态加载。
- 现在改为使用 `import.meta.glob('../../node_modules/naive-ui/es/*/index.mjs')`，由 Vite 在构建期生成模块 loader 映射。
- 保留原有 `getNaiveComponent('NInput')` API，现有 JSON 配置不用改。

说明：

- 这个改法避免了完全不可分析的字符串动态 import。
- 构建中仍然存在 Naive UI 动态导入和静态导入混用的 warning，因为项目其他地方仍然直接从 `naive-ui` 静态 import 组件。

### 4. 移除 `FormEditorButton` 中的 `eval`

修改文件：

- `src/components/FormEditorButton.tsx`

内容：

- 移除了通过 `eval` 把字符串还原为函数的逻辑。
- JSON 配置现在不支持用户手写函数源码。
- 代码中原本存在的函数会在编辑器里显示为 `__function_slot__x` 占位符。
- 保存时只允许将已有占位符还原成原函数，不执行用户输入的函数源码。

原因：

- JSON 应保持为纯数据。
- 避免任意 JS 执行风险。
- 为未来 schema 校验、配置保存、可视化编辑做准备。

### 5. 拆分编辑态能力和普通渲染能力

修改文件：

- `src/components/Form.tsx`
- `src/components/SearchFrom.tsx`
- `src/components/CustomTable.tsx`
- `src/components/FormEditorButton.tsx`
- `src/__tests__/admin.spec.ts`

内容：

- `Form`、`SearchFrom`、`CustomTable` 不再静态 import `FormEditorButton`，改为 `defineAsyncComponent`。
- `FormEditorButton` 不再静态 import `CardCodeEditor`，改为异步加载。

结果：

### 6. 修复路由切换时右侧滚动条问题

修改文件：

- `src/assets/main.css`

内容：

## 当前遗留问题

### 1. Naive UI 动态/静态导入混用 warning

构建中仍然有很多类似 warning：

- 某些 Naive UI 模块既被 `dynamicComponent.ts` 动态导入，又被项目或 Naive UI 内部静态导入。

这不是本轮改动引入的新错误，但后续可以优化。

可能方向：

- 统一 Naive UI 组件使用方式。
- 减少从 `naive-ui` 静态 import 大量组件。
- 或者调整动态组件解析策略，只动态加载表单配置里允许的组件集合。

### 2. 构建 chunk 仍然偏大

构建中仍有超过 500 kB 的 chunk warning。

已改善的地方：

- `FormEditorButton` 和 `CardCodeEditor` 已拆成独立 chunk。

后续可继续做：

- 路由级代码拆分。
- 图表库 ECharts 按页加载。
- CodeMirror 只在编辑态加载。
- 手动配置 `manualChunks`。

### 3. 路由测试还有 mock route warning

测试中 Vue Router 会提示 mock 路由缺少 component/children。

这不影响测试通过，但可以整理。

可能方向：

- 给 router 测试里的 mock route 补空组件。
- 或将路由守卫逻辑抽成可单测函数，减少对真实 router 的依赖。

### 4. 类型系统仍偏松

当前 `FormItem`、`tableButtonItem`、`searchButtonItem` 仍有不少 `any` 和 `Record<string, any>`。

后续应该逐步收紧：

- 抽出 `src/types/schema.ts`。
- 将表单 item、按钮 item、表格列扩展定义成稳定类型。
- 为 ProTable 做更清晰的泛型或数据适配类型。

## 重要设计共识

- JSON/schema 配置应保持纯数据，不允许用户配置函数源码。
- 自定义渲染能力应通过代码侧注册表或已有函数占位符实现。
- 普通渲染能力和编辑态配置能力要拆开。
- `基本表格` 保留为基础组件示例。
- `ProTable` 用作高级后台业务表格能力的主战场。
- 项目短期目标是先做稳定、可复用的 Naive UI Admin 底座，再向 JSON 配置和可视化配置演进。
